# Data Model: Reservation Context Storage

## Entity: Reservation Context

Represents the latest actionable reservation snapshot for one WhatsApp user.

### Fields

| Field                     | Type     | Required | Notes                                                    |
| ------------------------- | -------- | -------- | -------------------------------------------------------- |
| `id`                      | uuid     | yes      | Internal identifier generated by storage                 |
| `waId`                    | string   | yes      | WhatsApp user identifier; unique                         |
| `phone`                   | string   | yes      | Normalized operational phone used to locate reservations |
| `reservationDate`         | string   | yes      | Business-facing reservation date value                   |
| `reservationTime`         | string   | yes      | Business-facing reservation time value                   |
| `reservationStartsAt`     | datetime | yes      | Absolute start moment for active/expired decisions       |
| `reservationEndsAt`       | datetime | yes      | Absolute end moment for active/expired decisions         |
| `name`                    | string   | yes      | Customer name tied to the reservation                    |
| `quantity`                | integer  | yes      | Party size; must be positive                             |
| `status`                  | enum     | yes      | `active`, `cancelled`, or `expired`                      |
| `lastConversationSummary` | string   | no       | Optional short summary; never a full transcript          |
| `createdAt`               | datetime | yes      | Creation timestamp                                       |
| `updatedAt`               | datetime | yes      | Last update timestamp                                    |

### Constraints

- There must be at most one row per `waId`.
- `waId`, `phone`, `reservationDate`, `reservationTime`, and `name` must not be blank.
- `quantity` must be a positive integer.
- `reservationEndsAt` must be later than `reservationStartsAt`.
- `status` must be one of `active`, `cancelled`, or `expired`.
- `lastConversationSummary` is optional and must not contain full conversation history.

### Indexes

- Unique index on `waId`.
- Non-unique index on `status` and `reservationEndsAt` for active lookup and future expiration work.

## State Transitions

```text
new context -> active
active -> active      (new snapshot for same waId replaces previous values)
active -> cancelled   (context is no longer usable)
active -> expired     (reservation ended before cutoff)
cancelled -> active   (new snapshot for same waId replaces previous cancelled context)
expired -> active     (new snapshot for same waId replaces previous expired context)
```

## Validation Rules

- Active lookup returns a context only when:
  - `waId` matches the requested WhatsApp user.
  - `status` is `active`.
  - `reservationEndsAt` is greater than the comparison time.
- `markCancelledByWaId` affects only the matching user's current row if one exists.
- `markExpiredBefore` affects only rows with `status = active` and `reservationEndsAt` before the cutoff.
- Upsert always writes `status = active` for the latest saved snapshot.

## Relationships

No foreign keys are required in Phase 1.

`Reservation Context` intentionally has no `accountId` relationship and no relation to billing tables. Future multi-account support can introduce a scoped uniqueness rule if the reservation domain becomes multi-tenant.
